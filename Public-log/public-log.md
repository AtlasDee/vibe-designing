## 2026-07-14 (25) — 扩展支持前台 ToC 产品需求：新增"后台 vs 前台"两类视角

此前套件几乎只服务后台/中台（OMS/运营平台）需求；从一篇 C 端前端需求详细说明中学习前台 ToC 产品的书写方式，给 5 个 skill 各加一块"前台视角"，使其能同时梳理后台与前台 ToC（C 端页面/App/H5）产品需求：

- **总入口**：新增「产品视角：后台/中台 vs 前台 ToC」——两类产品在核心动作/状态机/特有表/字段与交互重心/决策归属上的对照；沉淀一份**前台全局通用原则**（纯展示不决策、无兜底则隐藏、登录拦截、人群/合规过滤、接口异常隔离、时间过滤），供各分件复用。
- **feature（规则层）**：明确权限/日志表是后台专属；前台改用「有效性过滤链」（一条有序过滤规则+明确隐藏或空状态）、动效行（参数写全）、状态-呈现对照矩阵（前台版状态矩阵）；前台不把排序/推荐/兜底写成前端职责。
- **field（字段表）**：前台字段的数据来源分五类（后台配置/后端接口/前端本地/前端计算/常量）、配置侧与展示侧两张表并列、展示字段的货币/脱敏/省略/角标/涨跌/时间等格式约定、硬枚举值域收全。
- **uiux（图与交互）**：前台交互说明重心转向动效/自适应/空态异常态/登录拦截；动效态与多态按状态给图。
- **产品方案层**：前台全局通用原则在方案层统一声明一次，各模块不重复。

未改动 Why 层与两个姊妹分件。5 个 skill 已重新打包并复扫，确认无敏感信息残留。

## 2026-07-13 (23) — 融入又一轮实战经验：云文档发布图片坑 + 全局交互概览 + 7 个 skill 补丁

从一轮批量修订与云文档局部发布对齐的实战中提炼经验，落进 7 个 skill：

- **publish（发布分件）**：修复云文档图片发布的头号坑——替换图片时必须显式带真实宽高，否则云端偶发不回填尺寸导致图片被压成方框、留白变形；SVG 需先转成 PNG 再上传；只改图注不必重传。新增「发布前必做双向 diff」的硬规定，防止全量覆盖抹掉云端已有的业务调整。新增多章节批量发布的顺序规则，避免相互错位。
- **uiux（图与交互说明）**：图注惯例做了一次修正——改回不写"来源"标注（更贴合团队长期习惯）；新增"全局交互概览"这类跨模块整体说明的标准写法（示意图+真实截图故事板双图并存，设计语言对齐已安装的 UI/UX 设计智能 skill，兼顾无障碍对比度）。
- **businessflow（产品方案）**：明确"精简名词解释"的两种不同操作（压缩措辞 vs 删除词条），避免歧义。
- **describe（摘要与日志）**：提醒插入变更日志表格行时的一个常见编辑陷阱。
- **field（字段表）**：任何清单类需求都必须套用标准字段表格式，不允许自造简化版。
- **feature（规则说明）**：新增"配置与运行时分开写、互相引用"的组织原则；新需求与既有内容冲突时先标记待确认、不擅自替用户决定；字段表在特定情况下可省略。
- **总入口**：补充"任何编号/结构调整都要全篇核查交叉引用"的通用约定。

`prototype` 分件本轮无对应经验，未改动。7 个 skill 已重新打包并复扫，确认无敏感信息残留。

## 2026-07-07 (22) — 融入一轮完整实战经验：原型回流列为强化情形 + 6 个 skill 补丁

从一次「BRD→初稿 PRD→交互原型→原型回流对齐 PRD→发布云文档」的完整实战中提炼经验，落进 6 个 skill：

- **总入口**：明确「只读 BRD 生成的初稿必然与后续原型产生偏差」，起草时即登记原型位置作为回流锚点；把「原型回流」（原型比 PRD 新时反过来修正 PRD）列为情形3 的典型强化场景，不因 PRD 表面完整就跳过检查。
- **feature（规则层）**：需求概述收紧为「谁·子功能·效果」三件事；改名类修正要求全文档 grep 检查残留；新增指标口径分层指引（原子埋点/派生计算/后端配置态计数三类，明确哪类不属于埋点范围）。
- **field（字段表）**：字段表须贴合原型的真实控件，原型精简/合并字段时 PRD 同步跟随；列表里每个可见字段（含计算类指标）都必须单独成行。
- **uiux（图与交互说明）**：原型迭代后旧图必须重截替换，不沿用过时界面；交互态截图方法优化为更省文件的单变体+参数化写法；交互概览新增非线性场景下的分支/网格布局画法。
- **prototype（原型工程）**：新增硬规定——PRD 里的内部规划标注不得出现在原型界面上；补充两条 CSS 常见陷阱与一份常见交互反馈自检清单。
- **publish（发布分件）**：把「只更新某一章节」列为一等情形并给出标准操作流程（备份→按标题定位区间→删除→按位置插入→复用原有转换逻辑→核对验证）；明确任何一次发布（含局部发布）都必须同步更新云文档自身的变更记录。

6 个受影响的 skill 已重新打包并复扫，确认无敏感信息残留。

## 2026-07-02 (20) — uiux 全屏截图改用 4K 分辨率，修正原型变形问题

- 根因：全屏截图窗口过窄会命中原型自身的响应式断点，导致桌面布局被压缩折叠成窄屏样式（"变形"），与清晰度倍数（DPR）无关。
- 修复：主功能模块的全屏图（交互概览取材）现要求 `--window-size=3840,2160`（4K）截取，不额外叠加 DPR 倍数；子功能模块的局部图不受影响，维持原有窗口尺寸+2×高清+自动裁白。内联组件的局部图统一从同一张 4K 全屏图裁剪得到。

## 2026-07-01 (11) — 三条硬规定：模块不合并 / 复杂多图 / 字段列不精简

针对实跑 PRD 暴露的三问题，强化三个子技能：

- **feature（模块粒度）**：一个组件/独立配置面板/独立抽屉 = 一个独立功能模块；⛔ 禁止用 `(1)(2)…` 把多个组件揉进一个模块、禁止合并两个不同组件。
- **uiux（图的覆盖）**：每个功能模块必须至少 1 张 UI 图；**复杂/多态模块必须按状态多截图**（条件显隐/弹窗抽屉/分支进度各态各一张，从"可多张"改为硬要求）；每个复杂大功能必须有交互概览故事板。
- **field（字段列）**：必须用完整列模板（搜索/编辑 11 列、列表 6 列）；⛔ 禁止自造精简列子集/删列；空值填 `-`；数据来源/提示文案/反馈/示例/报错文案不得省略。
- 三技能自检清单同步增补。

## 2026-06-30 (8) — uiux 截图质量：高清 + 自动裁白 + 图标加载

- **高分辨率**：截图加 `--force-device-scale-factor=2`（2× 高清）。
- **图标/字体加载**：加 `--virtual-time-budget`，等图标字体加载完再截，避免占位框。
- **自动裁白**：截后用 PIL `ImageChops.difference + getbbox` 裁掉四周纯色边，小卡片页只留元素。

## 2026-06-30 (7) — vibe-designing-uiux 经验固化

- **来源铁律**：UI 图只用所提供的交互原型截图，同项目来源统一；未指定来源一律不用。
- **截图方式**：首选无头浏览器 `--screenshot` 直接落盘；交互态用「注入自动展开脚本 + virtual-time-budget」的临时副本页，截完即删。浏览器扩展类截图可能不回传可用磁盘路径，不据此嵌入文档。
- **双重复查**：截图后逐张复查（真图/尺寸/内容正确）；嵌入后复查（路径/显示/图注相符）。

- 新增**输入来源优先级（不只依赖原始需求）**：现有 PRD（基准）＞ 交互原型/项目资料（落地，常修正原始需求）＞ 原始需求（方向）。
- 四种情形/输入原则贯穿到 功能清单/功能模块/需求概述/字段说明/功能说明 各产出。

## 2026-06-30 (2) — vibe designing skill 立项与构建

- 确立目标：产品文档整理技能（业务需求 → 产品方案 → 结构规范的 PRD）。
- 构建「1 入口 + 4 分件」共 5 个 skill：
  - 总入口/编排
  - describe（摘要/版本/变更日志/背景期望）
  - businessflow（方案目标/名词/流程/架构/功能清单/Out-of-Scope）
  - field（搜索/编辑/列表三类字段表，含新增的编辑配置字段表）
  - feature（需求概述/功能说明表/状态矩阵/权限/日志）
- 设计依据：归纳 PRD 写作 house style + 需求文档模板，吸收通用 PM 方法论（摘要句式、成功指标三件套、显式 Out-of-Scope、🔶🔵 行内缺口标记、产出自检）。
- 旧的两个 skill 被升级取代；各 skill 以 `.skill`（ZIP，内含 `SKILL.md`）打包。
